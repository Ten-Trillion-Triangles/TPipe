# PipeContextProtocol Package API

The Pipe Context Protocol (PCP) is the Tool Belt of the TPipe ecosystem. It provides a comprehensive, high-security framework that allows AI models to interact with the external world through multiple transport mechanisms—including local shell commands (Stdio), remote APIs (HTTP), sandbox scripting (Python), and native application logic (TPipe functions).

```kotlin
@Serializable
data class PcpContext(@Transient val cinit: Boolean = false)
```

## Core Classes

### PcpContext
The main configuration reservoir for PCP. It defines the availability and security settings for all tools accessible to an agent.
- **`stdioOptions` / `httpOptions` / `tpipeOptions`**: Lists of the specific tool blueprints attached to the belt.
- **Security Boundaries**: `allowedDirectoryPaths` and `forbiddenFiles` provide filesystem-level protection, ensuring tools cannot escape their designated workspace.

### PcPRequest
 The cargo object generated by an LLM to request a tool call. It contains the specific context options for the chosen transport and the actual parameters (e.g., shell arguments or script content) for execution.

### PcpExecutionDispatcher
The central Switching Station for tools. It determines the transport type, verifies the request against the security boundaries in the `PcpContext`, and routes the call to the appropriate executor.

### PcpResponseParser
A high-resilience parser that extracts tool requests from raw model output. It uses TPipe's automated JSON repair to handle minor formatting errors in the model's response.

## Context Options: The Fittings

Each tool type has its own specialized configuration "Fitting."

#### StdioContextOptions (Shell Tools)
- **`command`**: The binary or script to execute.
- **`executionMode`**: Supports `ONE_SHOT` for simple calls or `INTERACTIVE` for persistent terminal sessions.
- **`bufferPersistence`**: If true, TPipe records the entire session flow for later auditing or replay.

#### HttpContextOptions (Network Tools)
- **`baseUrl` / `endpoint`**: The coordinates for the remote API.
- **`authType`**: Supports `BASIC`, `BEARER`, and `API_KEY` for secure connections.
- **`allowedHosts`**: A strict allow-list used for SSRF protection.

#### PythonContext (Scripting Tools)
- **`pythonPath`**: The location of the interpreter.
- **`availablePackages`**: An allow-list of modules (e.g., `pandas`, `json`) the model is permitted to import.

#### TPipeContextOptions (Native Logic)
- **`functionName`**: The identifier used by the model to call your Kotlin code.
- **`params`**: A map of parameter definitions including types and descriptions.

## Result and Validation Classes

#### PcpRequestResult
The diagnostic report for a single tool call, containing the `success` status, the raw `output` string, and the `executionTimeMs`.

#### PcpValidationResult
A pre-flow report. TPipe uses this to ensure a request has all required fields (like a valid URL for HTTP or an existing command for Stdio) before it ever reaches the dispatcher.

## Key Operational Behaviors

### 1. Multi-Layer Security
PCP implements industrial-grade "Leash" logic. Security is enforced at the individual tool level (permissions), the transport level (host allow-lists), and the system level (filesystem restrictions).

### 2. Parallel Dispatch
When a model requests multiple tools in a single turn, the `PcpExecutionDispatcher` can process them concurrently using Kotlin Coroutines, maximizing throughput and reducing total latency.

### 3. Robust Fault Tolerance
The `PcpResponseParser` is designed for the "Real World." It uses bracket matching and lenient parsing to extract valid tool calls even if the model has surrounded them with conversational text or made minor syntax errors.

### 4. Deterministic Identity
Every tool call is tagged with a `Transport` enum and its associated metadata, ensuring that your telemetry logs (Tracing) provide a clear audit trail of exactly how the agent interacted with your infrastructure.
